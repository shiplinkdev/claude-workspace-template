# grill: Design Interview → PM Handoff

## Purpose

Conduct a methodical, one-question-at-a-time interview to stress-test a feature plan against your team's domain model and documented decisions. Resolve all ambiguity, then produce a structured PM Handoff Summary.

**Do not create GitHub issues. Do not write a PRD. Produce resolved context for the PM Agent.**

## Trigger

User invokes `/grill` with a feature idea, problem description, or GitHub issue number.

## Before Starting

Read these files to ground the session:
- `CONTEXT.md` — domain glossary, role definitions, key constraints
- `docs/adrs/` — existing architecture decisions (read index, open relevant ones)
- `.claude/rules/structure.md` — where code lives
- `.claude/rules/stack.md` — tech constraints

Load additional rule files when the feature touches those areas (see the table in `CLAUDE.md`).

If a GitHub issue number or URL is given, resolve it:
- Bare number `#123` → assume current repo (`TODO: your-org/your-repo`)
- Full URL `https://github.com/<org>/<repo>/issues/<number>` → extract org, repo, and number
- Fetch: `gh issue view <number> --repo <org>/<repo> --json title,body,labels,assignees,comments`

When the issue is in a different repo, note it in the handoff so the PM Agent knows which codebase is affected.

## Interview Rules

- Ask **one question at a time**. Wait for the answer before continuing.
- Provide a **recommended answer** with each question so the user can accept or redirect.
- Explore the codebase when the answer might already exist there — don't ask what you can discover.
- **Challenge terminology** — flag when user language conflicts with `CONTEXT.md` glossary.
- **Test with scenarios** — probe edge cases through concrete examples to expose boundary assumptions.
- **Verify against code** — surface contradictions between stated behavior and implementation.
- Walk dependency trees: resolve upstream decisions before downstream ones.

## What to Resolve

Cover all of these before producing the handoff:

1. **Problem** — what user pain or gap are we solving, and for which user type?
2. **Scope boundary** — what is explicitly out of scope?
3. **Domain fit** — does this interact with existing domain states, real-time events, or role permissions?
4. **Technical constraints** — any conflicts with existing ADRs or stack rules?
5. **Acceptance shape** — what does "done" look like from a user perspective?
6. **Open PM questions** — what requires business/product decisions, not technical ones?

## Capture Decisions Immediately

- When a term resolves, update `CONTEXT.md` glossary inline.
- When a hard-to-reverse, surprising architectural decision emerges from the interview, create an ADR in `docs/adrs/ADR-NNN-short-description.md`.
- Create ADRs conservatively — only for genuine trade-offs that future readers would not infer.

## Output: PM Handoff Summary

When all questions are resolved, produce this block:

```markdown
## PM Handoff: <feature name>

**Issue**: <repo>#<number> | (no issue — session only)
**Interviewed by**: grill session on <date>

### What we're building
[1–2 sentences, decided and precise]

### Why
[Motivation, user pain, and constraints that shaped direction]

### Domain decisions made
- [Specific decisions — reference CONTEXT.md terms and ADR numbers where relevant]

### Acceptance criteria (draft)
- [ ] ...
- [ ] ...

### Open questions for PM
- [Items requiring business/product decisions, not technical ones]

### Do NOT do
- [Explicit scope exclusions decided during the session]

---
*Generated by `/grill` — run `/to-prd` (local or PM Agent) to produce the full PRD.*
```

### If a GitHub issue was given

Post the handoff as a comment on that issue:

```bash
gh issue comment <number> --repo <org>/<repo> --body "<handoff markdown>"
```

Then output the comment URL so the user can share it with the PM Agent.

### If no GitHub issue exists

Output the handoff block in the conversation only. The user works from the session directly — they can run `/to-prd` locally or copy the block into the PM Agent manually. Do not create an issue automatically.
